# protocols — Day 1

## Q1: How would you approach selecting a communication protocol for a new battery-powered medical sensor device that needs to stream data to a central monitor?
**Answer:** I’d start by defining the key constraints: data rate required, distance between sensor and monitor, power budget, number of nodes, and any regulatory or isolation requirements for medical safety. For a battery-powered device, low-power operation is critical, so protocols like I2C or SPI are common for on-board sensor communication, but for external streaming, wireless options like BLE or a wired differential pair (RS485 or CAN-FD) might be more appropriate. I’d evaluate trade-offs: BLE offers low power and cable-free convenience but has latency and coexistence concerns in a clinical environment; RS485 is robust over longer distances and inherently differential for noise immunity, but adds cabling. I’d also consider whether the protocol supports the required data integrity (e.g., CRC, acknowledgment) and whether it can be implemented with the chosen microcontroller’s peripherals to avoid extra components. Finally, I’d prototype a small-scale test to measure actual power consumption and error rates under realistic conditions before committing.

**Possible follow-ups:** How would you handle the case where the sensor data rate exceeds the protocol’s throughput? What isolation requirements might apply for a patient-connected device using a wired protocol?

## Q2: You’re debugging a system where an I2C bus intermittently hangs, causing the sensor to stop responding. How would you approach this?
**Answer:** I’d start by reproducing the issue with a logic analyzer or oscilloscope to capture the bus signals—SCL and SDA—during the hang. Common causes include a slave device holding SDA low (clock stretching issue or bus lock-up), a missing pull-up resistor or incorrect value, or a glitch on the clock line. I’d check the waveform for proper start/stop conditions, acknowledge bits, and timing relative to the datasheet specs. If a slave is holding the line low, I’d look for a stuck state in its firmware or a power sequencing problem. I’d also verify the pull-up resistor value is appropriate for the bus capacitance and speed (e.g., 4.7 kΩ for standard mode, lower for fast mode). If the issue is intermittent, I might add a bus recovery routine in firmware—toggling SCL until the slave releases SDA—or implement a timeout and reset mechanism. Finally, I’d review the PCB layout for crosstalk or excessive trace length that could degrade signal integrity.

**Possible follow-ups:** How would you distinguish between a hardware fault and a firmware bug causing the hang? What if the bus has multiple slaves—how would you isolate which one is causing the problem?

## Q3: How would you design a firmware layer to handle multiple communication protocols (e.g., I2C, SPI, UART) on a single microcontroller in a modular, testable way?
**Answer:** I’d use an abstraction layer with a common interface—for example, a struct of function pointers for init, read, write, and deinit—so the higher-level application code doesn’t depend on the specific protocol. Each protocol driver would implement that interface, handling the low-level register access and interrupt or DMA configuration. I’d also separate the transport layer (raw byte transfer) from the application layer (message framing, error checking, retry logic) to make each part independently testable. For testing, I’d create mock drivers that simulate the protocol behavior, allowing unit tests to verify the application logic without hardware. I’d also include a configuration table that maps logical endpoints (e.g., “temperature sensor”) to a protocol instance and pin assignment, so changing a sensor only requires updating the table, not rewriting code. This approach keeps the system maintainable and portable across microcontrollers.

**Possible follow-ups:** How would you handle priority or arbitration if two protocols need to share a single DMA channel? What testing strategy would you use to verify the abstraction layer doesn’t introduce latency?

## Q4: In a medical device using CAN-FD for communication between subsystems, you encounter intermittent message loss under high bus load. How would you debug this?
**Answer:** I’d start by capturing bus traffic with a CAN analyzer to see if the lost messages are due to arbitration (lower-priority messages being delayed), bit errors (CRC or stuff-bit errors), or buffer overruns at the receiver. I’d check the bus termination—improper termination can cause reflections and bit errors at high speeds. I’d also verify the CAN-FD bit timing configuration (nominal and data phase) matches the network topology and cable length. On the firmware side, I’d examine the transmit and receive mailbox configurations: are there enough mailboxes to queue messages, and is the ISR handling fast enough to avoid overruns? I’d also look at the message filtering—if filters are too restrictive, valid messages might be discarded. If the issue is arbitration-related, I might adjust message priorities or reduce the bus load by batching data. Finally, I’d add diagnostic counters (e.g., error frames, bus-off events) to the firmware to log during operation for post-mortem analysis.

**Possible follow-ups:** How would you decide between increasing the CAN-FD data rate versus reducing the number of messages? What safety considerations apply when changing CAN bus parameters in a medical device?

## Q5: Imagine you’re leading a cross-functional team where a firmware engineer and a hardware engineer disagree on whether to use SPI or I2C for a new sensor interface. How would you handle the disagreement?
**Answer:** I’d first facilitate a structured discussion where each engineer presents their technical rationale—SPI might be preferred for higher data rates and simpler timing, while I2C might be chosen for fewer pins and multi-slave capability. I’d ask them to list the specific requirements: data rate needed, distance on the PCB, number of sensors, power constraints, and any existing IP or library support. Then I’d propose we evaluate both options against those requirements with a simple trade-off matrix, possibly prototyping both on a breadboard to measure real-world performance (e.g., signal integrity at the required speed, power consumption). If the disagreement persists, I’d make a decision based on the data and the project’s priorities—for example, if reliability under noise is critical, SPI’s separate clock and data lines might win; if pin count is limited, I2C might be better. I’d ensure both engineers understand the reasoning and document the decision, including any mitigation for the chosen protocol’s weaknesses. The goal is to turn the disagreement into a productive exploration of trade-offs, not a win-lose argument.

**Possible follow-ups:** How would you handle it if one engineer’s preferred protocol introduces a schedule risk due to lack of driver maturity? What if the disagreement is actually about personal preference rather than technical merit—how do you detect that?
