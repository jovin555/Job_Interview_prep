# high-speed-digital-fpga — Day 1

## Q1: How would you approach selecting a clock distribution scheme for a high-speed FPGA design with multiple clock domains operating at different frequencies (e.g., 200 MHz, 400 MHz, and a 1.25 Gbps transceiver reference)?

**Answer:** The first step is to analyze the clocking requirements of each domain: the frequency, jitter tolerance, and whether the clocks need to be phase-aligned or can be asynchronous. For a design like this, I'd typically start with a low-jitter crystal oscillator or MEMS oscillator as the primary reference, then use the FPGA's internal PLLs/MMCMs to generate the derived frequencies. The 1.25 Gbps transceiver reference is the most sensitive — it usually requires an ultra-low-jitter source (often a dedicated oscillator or a clean PLL output) because transceiver clock jitter directly impacts bit-error rate. For the 200 MHz and 400 MHz domains, I'd check if they can be generated from the same PLL to maintain a known phase relationship, which simplifies clock-domain-crossing (CDC) analysis. I'd also plan for dedicated clock input pins (MRCC/SRCC on Xilinx, or equivalent) to minimize skew, and avoid routing clocks through general-purpose fabric. Finally, I'd simulate the clock tree in the FPGA tool's timing analyzer to verify setup/hold margins across process, voltage, and temperature corners.

**Possible follow-ups:** How would you handle a situation where the PLL cannot lock across the full temperature range? What clock jitter specifications would you request from the oscillator vendor?

## Q2: How would you debug a signal integrity issue on a high-speed parallel bus (e.g., DDR3/DDR4 memory interface) that fails timing closure only at high temperature?

**Answer:** This sounds like a temperature-dependent signal integrity problem, which often points to timing margin erosion rather than a hard failure. I'd start by collecting detailed characterization data: run the memory interface at multiple temperatures and voltage corners, and use the FPGA's built-in calibration and read/write leveling features to see how the delay taps shift. If the FPGA has a debug interface like Xilinx Vivado's ILA or Intel's Signal Tap, I'd capture actual data eye patterns at the failing temperature. Common culprits include: (1) on-die termination (ODT) values that are mismatched with the PCB trace impedance, where the impedance changes with temperature; (2) DQS-to-DQ skew that drifts because of uneven thermal expansion or different trace lengths; (3) the PLL or DLL that generates the memory clock becoming unstable at high temperature. I'd also check the power supply ripple on the memory VDD/VREF rails at temperature — a regulator with poor thermal performance can droop and reduce timing margin. The fix might involve adjusting the output drive strength, changing ODT settings, or adding a small amount of delay on individual DQ lines through the FPGA's IODELAY elements. If the issue persists, I'd review the PCB stackup and ensure the memory traces have proper ground referencing and no impedance discontinuities near vias or connectors.

**Possible follow-ups:** What specific DDR timing parameters would you measure first? How would you distinguish between a signal integrity issue and a power integrity issue?

## Q3: How would you approach designing a board-to-board high-speed interconnect (e.g., 10 Gbps differential pairs) between two PCBs connected through a mezzanine connector?

**Answer:** I'd treat this as a controlled-impedance transmission line problem from end to end. First, I'd select a connector rated for the target data rate — many standard mezzanine connectors have specified insertion loss and return loss up to 10+ Gbps, but I'd verify the manufacturer's S-parameter models. The PCB stackup on both boards needs to support differential impedance (typically 100 ohms differential) with a consistent reference plane. I'd keep the differential pairs as short as possible, avoid vias in the high-speed path, and if vias are unavoidable, I'd use back-drilling or at minimum ensure the via stub is minimized. On the connector side, I'd follow the manufacturer's recommended footprint and routing pattern — many connectors have specific pin assignments for high-speed signals to minimize crosstalk. I'd also simulate the entire channel (driver → PCB trace → connector → PCB trace → receiver) using the S-parameter models in a tool like HyperLynx or ADS to check the eye diagram at the receiver. If the link margin is tight, I might add pre-emphasis or equalization at the transmitter, or use a receiver with CTLE (continuous-time linear equalization). Finally, I'd include test points or a loopback mode in the FPGA firmware to allow BER (bit-error rate) testing during board bring-up.

**Possible follow-ups:** How would you decide whether to use AC-coupling capacitors on these differential pairs? What crosstalk mitigation techniques would you apply in the connector region?

## Q4: How would you approach verifying that an FPGA design meets its timing requirements when the design uses multiple asynchronous clock domains?

**Answer:** The key is to properly constrain all clock domains and then systematically analyze the clock-domain crossings (CDCs). I'd start by writing a complete SDC (Synopsys Design Constraints) file that defines every clock, its period, waveform, and any generated clocks from PLLs. For asynchronous crossings, I'd use the `set_false_path` constraint for paths that are truly asynchronous and have proper synchronization logic (e.g., dual-flop synchronizers, FIFOs with gray-code pointers). For CDC paths that are synchronous but in different clock groups, I'd use `set_clock_groups -asynchronous` to tell the tool not to analyze those paths. Then I'd run both static timing analysis (STA) and a CDC verification tool (like Mentor's Questa CDC or Synopsys SpyGlass CDC) to catch metastability issues or missing synchronizers. I'd also run a formal equivalence check if the design was modified after synthesis. For the STA report, I'd focus on the worst negative slack (WNS) and total negative slack (TNS) — if any path fails, I'd examine whether it's a real constraint violation or a mis-constrained false path. Finally, I'd run a gate-level simulation with back-annotated delays to verify the design works under worst-case conditions, especially for the CDC paths that STA treats as unconstrained.

**Possible follow-ups:** What synchronizer depth would you use for a clock crossing between a 200 MHz domain and a 50 MHz domain? How would you detect a missing synchronizer in a CDC path?

## Q5: Behavioral question — You're leading a design review for a high-speed FPGA-based data acquisition board. A junior engineer on your team proposes using a single-ended clock distribution scheme for a 500 MHz clock, arguing it saves board space and cost. How would you handle this situation?

**Answer:** I'd start by acknowledging the engineer's cost-saving intent — that's a good engineering instinct. Then I'd walk through the technical trade-offs collaboratively rather than dismissing the idea outright. I'd explain that at 500 MHz, single-ended clock distribution is susceptible to common-mode noise coupling, ground bounce, and crosstalk from adjacent traces, which can introduce jitter that degrades the FPGA's timing margin. I'd ask the engineer to calculate the expected jitter budget: what's the oscillator's phase noise, what's the added jitter from the single-ended trace (using IBIS simulations or a simple transmission line model), and does the FPGA's clock input have enough jitter tolerance? I'd also point out that differential signaling (LVDS or LVPECL) inherently rejects common-mode noise and provides better signal integrity over longer traces. If the engineer can show that the single-ended approach meets the jitter budget with margin, I'd be open to it — but I'd also suggest adding a differential option as a footprint alternative on the PCB, so the design can be re-spun if testing reveals issues. The goal is to teach the trade-off analysis process while keeping the design robust. After the review, I'd follow up with a short whiteboard session on clock distribution topologies to build the team's knowledge.

**Possible follow-ups:** How would you quantify the cost difference between single-ended and differential clock distribution for this design? What if the engineer insists on single-ended because of a tight schedule — how would you decide whether to accept the risk?
