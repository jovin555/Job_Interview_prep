# embedded-linux-bsp — Day 1

## Q1: How would you approach bringing up a custom board running Embedded Linux for the first time?
**Answer:** I would start by verifying the hardware fundamentals before touching software — checking power rails, clock signals, and reset sequencing with an oscilloscope. Once those are stable, I'd begin with a minimal boot chain: typically U-Boot or Barebox as the bootloader, configured for the specific SoC and memory map. I'd use a known-good kernel defconfig (often the vendor's reference) and a minimal root filesystem (Buildroot or a ramdisk) to isolate variables. The bring-up proceeds incrementally: first get serial console output from the bootloader, then verify DDR initialization works, then load the kernel via TFTP or from SD card, and finally mount a rootfs. Each step is validated before moving to the next. If the board doesn't boot, I'd check for common issues like incorrect memory timings, missing pinmux configurations, or wrong device tree entries.

**Possible follow-ups:** How would you debug a kernel panic that occurs during early boot before the console is visible? What tools would you use to verify DDR timing parameters are correct for your specific board layout?

## Q2: What considerations go into selecting between Yocto and Buildroot for a medical IoT device project?
**Answer:** The choice depends on project requirements around customization depth, update strategy, and team expertise. Yocto offers more granular control over every package, build-time configuration, and layer management — which is valuable when you need strict version pinning for regulatory traceability, or when you're building for multiple hardware variants from a single BSP. It also has better support for generating custom SDKs for application developers. However, Yocto has a steeper learning curve and longer initial build times. Buildroot is simpler and faster to get started with, producing a smaller root filesystem, which can be advantageous for resource-constrained devices. For a medical device, I'd lean toward Yocto if the product requires long-term maintenance, security patching, and auditable build reproducibility — all of which align with regulatory documentation requirements. Buildroot might be appropriate for an early prototype or a simpler device where the team needs rapid iteration.

**Possible follow-ups:** How would you handle OTA update capability in either build system? What's your approach to maintaining a Yocto layer over a multi-year product lifecycle?

## Q3: How would you debug a device tree configuration that causes a peripheral (e.g., an I2C sensor) to fail to probe during kernel boot?
**Answer:** I'd start by checking the kernel boot log with `dmesg` for the specific error message — common ones include "failed to probe", "no matching node found", or "resource conflict". I'd verify that the device tree node has the correct compatible string matching the driver, proper reg address, and that the parent bus node is enabled and clocked. I'd also check that pin control (pinctrl) settings are correct for the I2C pins on that SoC. If the driver is built as a module, I'd confirm it's loaded and check `modinfo` for module parameters. For runtime debugging, I'd use `devicetree.dtb` overlays to test changes without rebuilding the entire kernel, and inspect the live device tree via `/proc/device-tree` or `dtc -I fs /proc/device-tree`. If the issue persists, I'd add debug prints to the driver or use `ftrace` to trace the probe function. A common gotcha is that the I2C bus itself may be working but the sensor's power supply or reset GPIO isn't being asserted correctly in the device tree — so I'd also check the regulator and GPIO-handle references.

**Possible follow-ups:** How would you handle a case where the device tree is correct but the kernel driver has a bug in its probe sequence? What's the difference between a device tree overlay and a full device tree blob?

## Q4: How would you approach writing a kernel driver for a custom SPI-based ADC that needs to sample at 1 kHz and push data to userspace?
**Answer:** I'd evaluate whether this needs a character driver or can be handled by the existing IIO (Industrial I/O) subsystem. For a simple ADC, IIO is usually the right choice because it provides a standardized interface, sysfs access, and trigger/buffer support. I'd implement a driver that registers with the SPI subsystem, uses the IIO framework's `iio_dev` structure, and sets up a triggered buffer. For 1 kHz sampling, I'd use a hardware timer trigger (from the SoC) rather than a software timer to avoid jitter. The SPI transfer would be done in the trigger handler using `spi_sync_transfer()` or `spi_async()` if DMA is available. Data would be pushed to the IIO buffer, which userspace can read via `iio_read()` or through the sysfs interface. I'd also implement basic power management — suspend/resume callbacks — since this is likely battery-powered in a medical context. For testing, I'd validate the sampling rate with a known input signal and check for dropped samples using the IIO buffer's watermark and threshold features.

**Possible follow-ups:** How would you handle the case where the SPI bus is shared with another device and you need to guarantee the ADC's timing? What considerations would you have for real-time data delivery to a userspace application?

## Q5: A junior engineer on your team has submitted a BSP change that breaks the boot on half the hardware variants. How would you handle this situation?
**Answer:** First, I'd ensure the broken boards are identified and blocked from receiving the update — either by reverting the change in the build system or by pinning the known-good BSP version for those variants. Then I'd work with the junior engineer to understand what they were trying to achieve and where the testing gap occurred. I'd walk through the debugging process with them: checking what changed in the device tree or kernel config, comparing the working vs. non-working boot logs, and identifying whether the issue is in the bootloader, kernel, or rootfs. I'd use this as a teaching moment about the importance of testing across all hardware variants, perhaps suggesting they set up a CI pipeline that boots each variant in QEMU or on actual hardware before merging. I'd also review our team's process — was there a missing review step? Were the variant-specific test cases documented? The goal isn't to assign blame but to strengthen the team's practices so this doesn't recur. Finally, I'd have the junior engineer write the fix themselves (with guidance) so they own the correction and learn from the experience.

**Possible follow-ups:** How would you structure a BSP test plan to catch variant-specific issues before merge? What's your approach to code review for device tree changes that affect multiple boards?
