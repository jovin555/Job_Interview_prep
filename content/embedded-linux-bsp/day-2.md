# embedded-linux-bsp — Day 2

## Q1: How would you approach bringing up a custom board with a new SoC that has no existing BSP from the silicon vendor?

**Answer:** I would start by identifying the closest reference platform — either an evaluation kit from the SoC vendor or a publicly available board with the same or similar SoC family. The first step is to get a bootable system using the vendor's SDK or Yocto/Buildroot layer for that reference board, even if it doesn't fully match our hardware. From there, I'd work through the bring-up in layers: first, ensure the bootloader (U-Boot or Barebox) can initialize the clock tree, DRAM controller, and serial console. This typically requires adjusting the device tree to match our board's memory map, pin multiplexing, and peripheral connections. Once U-Boot boots to a prompt, I'd move to the Linux kernel, starting with a minimal defconfig and a device tree that describes only the essential peripherals (UART, SD/MMC, Ethernet). I'd verify each peripheral one at a time, using loopback tests or simple drivers before enabling more complex subsystems. Throughout this process, I'd maintain a bring-up checklist and track which device tree nodes have been validated. If the SoC vendor provides no BSP at all, I'd look for mainline Linux support for the SoC family and use the kernel's device tree bindings documentation as my primary reference.

**Possible follow-ups:** How would you debug a kernel hang during early boot before the console is visible? What tools would you use to verify that the DRAM initialization is correct?

## Q2: You're debugging a custom board where the kernel boots but the Ethernet PHY is not detected over MDIO. How would you approach this?

**Answer:** I'd start by confirming the basics: is the MDIO clock running? I'd probe the MDC pin with an oscilloscope to verify there's a clock signal when the kernel tries to probe the PHY. If there's no clock, the issue is likely in the device tree — either the MDIO bus node is missing, the pin muxing is incorrect, or the clock gating for the Ethernet MAC is not enabled. If the clock is present, I'd check the MDIO transactions using a logic analyzer to see if the PHY is responding with its expected PHY ID. A common issue is that the PHY address in the device tree doesn't match the hardware strapping. I'd also verify the PHY's reset GPIO — if the PHY is held in reset, it won't respond. I'd check the bootloader logs to see if the PHY was probed there (U-Boot often does a PHY probe during initialization). If U-Boot sees the PHY but Linux doesn't, the device tree may have a different MDIO bus node structure. I'd also check the kernel's PHY driver binding — some PHYs need a specific driver or a compatible string in the device tree. Finally, I'd enable dynamic debugging for the PHY subsystem (`CONFIG_PHYLIB_DEBUG`) and the MAC driver to get verbose probe logs.

**Possible follow-ups:** What if the PHY is detected but link never comes up? How would you distinguish between a hardware issue (bad magnetics) and a software issue (incorrect PHY configuration)?

## Q3: How would you structure a Yocto layer for a product that supports two hardware variants sharing the same SoC but with different peripheral sets?

**Answer:** I'd create a layered structure that maximizes reuse. The base layer would contain the common BSP components: the kernel recipe (with a defconfig that covers both variants), the bootloader recipe, and any shared firmware blobs. This layer would also include a base machine configuration file that defines the common SoC family and tuning. Then I'd create two machine-specific layers (or machine configs within the same layer) — one per hardware variant. Each machine config would inherit the base and override the device tree selection, kernel configuration fragments (using `.cfg` files to enable/disable variant-specific drivers), and any variant-specific recipes (e.g., a different WiFi module driver). For the device tree, I'd maintain separate `.dts` files for each variant in the kernel recipe, and the machine config would set `KERNEL_DEVICETREE` appropriately. If there's shared logic (e.g., a common base device tree that both variants include via `#include`), I'd put that in the kernel tree alongside the variant-specific files. For user-space differences (e.g., different init scripts for different hardware), I'd use `MACHINE` overrides in the recipes or bbappends. This approach keeps the build system clean — a developer just sets `MACHINE = "variant-a"` and gets the right image.

**Possible follow-ups:** How would you handle a situation where a driver needs a different kernel configuration for each variant? What if the variants share 90% of the same kernel config?

## Q4: A junior engineer on your team has written a kernel module that works on their development board but crashes on the production hardware. Walk through how you would guide them through the debugging process.

**Answer:** First, I'd ask them to confirm the crash is reproducible and capture the full kernel oops or panic message, including the register dump and backtrace. The first thing to check is whether the crash happens during module initialization, a specific operation, or is intermittent. I'd guide them to compare the two hardware platforms systematically: are the memory maps different? Does the module perform any direct memory-mapped I/O that might use wrong addresses or incorrect register offsets? I'd have them verify that the module's probe function properly handles deferred probing and that all required clocks, resets, and power domains are enabled. If the module uses DMA, I'd check whether the DMA mask is set correctly and whether the production hardware has different memory constraints (e.g., no IOMMU or different memory topology). I'd also have them check for endianness issues if the SoC families differ. Next, I'd suggest adding structured logging at each critical step and using `dev_dbg()` with dynamic debug enabled. If the crash is a NULL pointer dereference, I'd have them trace which pointer is NULL and work backward to find where it should have been initialized. I'd also recommend using `CONFIG_DEBUG_KERNEL`, `CONFIG_SLUB_DEBUG`, and `CONFIG_DEBUG_ATOMIC_SLEEP` to catch subtle bugs. Throughout, I'd emphasize the importance of understanding the hardware differences first — often the root cause is a timing difference or a missing hardware initialization step that the development board happened to have pre-configured by the bootloader.

**Possible follow-ups:** What if the crash only happens under heavy I/O load? How would you use ftrace or perf to narrow down the issue?

## Q5: Behavioral — You're leading the BSP development for a new product, and the project manager asks you to commit to a delivery date for a fully functional Linux boot. The SoC is new to your team, and you haven't received the first prototype boards yet. How would you handle this?

**Answer:** I'd explain that I can't give a single date with confidence, but I can provide a phased estimate with clear dependencies and risk factors. I'd break the work into stages: pre-silicon preparation (getting the toolchain, SDK, and reference BSP working on an evaluation board), first power-on (verifying the boot ROM and serial console), bootloader bring-up, kernel bring-up, and peripheral validation. For each stage, I'd give a best-case and worst-case timeline based on past experience with similar SoC families. I'd also identify the key unknowns: whether the SoC vendor's BSP is mature, whether the board's power sequencing matches the SoC requirements, and whether any critical peripherals use non-standard interfaces. I'd propose a checkpoint approach — after the first power-on, we'd have a much better estimate for the remaining stages. I'd also ask the PM what "fully functional" means: does it require all peripherals working, or just booting to a shell with networking? By clarifying the minimum viable definition, we might be able to deliver an earlier milestone. Finally, I'd suggest ordering an evaluation board from the SoC vendor in parallel with the custom board design, so we can start software development immediately and reduce the risk of discovering SoC-level issues late. This shows the PM that I'm thinking about risk mitigation, not just pushing back on the deadline.

**Possible follow-ups:** What if the PM insists on a single date for the project plan? How would you negotiate a buffer or contingency?
