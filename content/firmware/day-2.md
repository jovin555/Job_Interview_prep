# firmware — Day 2

## Q1: How would you approach designing a firmware architecture for a battery-powered medical sensor device that must run continuously for several days while maintaining reliable data capture?

**Answer:** I would start by identifying the key constraints: power budget, real-time data capture requirements, and safety-critical reliability. The architecture would center on a low-power microcontroller with multiple sleep modes. I'd use a real-time operating system (RTOS) like Zephyr to manage task scheduling, but carefully design the task structure to maximize time in deep sleep. The sensor sampling would be interrupt-driven rather than polled, with a dedicated low-power timer waking the system only when needed. Data would be buffered in a small circular buffer in RAM during active periods, then batch-processed or transmitted during brief wake cycles. I'd implement a state machine for power modes — active, idle, sleep, and deep sleep — with explicit transitions controlled by a power manager module. For reliability, I'd add a watchdog timer with a separate low-frequency oscillator, and use CRC checks on any stored data to detect corruption from low-voltage conditions. The firmware would also include brown-out detection to trigger an orderly shutdown before voltage drops below the operational threshold.

**Possible follow-ups:** How would you decide between using an RTOS versus a bare-metal scheduler for this kind of device? What strategies would you use to characterize the actual power consumption of your firmware early in development?

## Q2: You're debugging a firmware issue where a medical device intermittently fails to respond to a command sent over I2C, but only after it has been running for several hours. How would you approach this?

**Answer:** This sounds like a classic intermittent failure, likely related to timing, resource exhaustion, or a subtle state corruption. I'd start by trying to reproduce the failure under controlled conditions — logging the exact sequence of events leading up to the failure, including timestamps of all I2C transactions. I'd check whether the I2C bus is being left in a hung state (SCL low, SDA stuck) by examining bus signals with an oscilloscope or logic analyzer at the moment of failure. On the firmware side, I'd look for common causes: a missing timeout handler on the I2C transaction, a buffer overflow in the command queue, or a task priority inversion where a lower-priority task holding a mutex blocks the I2C handler. I'd add debug logging around the I2C state machine transitions and check if any interrupt service routines are taking too long, potentially missing I2C events. If the device has a real-time clock, I'd correlate the failure time with any periodic tasks that run on that schedule. I'd also examine whether the failure correlates with temperature changes inside the enclosure as the device warms up over hours of operation — that could indicate a marginal timing issue that only appears at certain temperatures.

**Possible follow-ups:** What tools would you use to capture the I2C bus behavior without disturbing the system? How would you distinguish between a hardware and firmware root cause for this kind of intermittent failure?

## Q3: How would you handle firmware updates (OTA or otherwise) for a Class II medical device that cannot be taken offline easily?

**Answer:** Firmware updates for medical devices require a fail-safe approach because an interrupted update could render the device inoperable. I would design a dual-bank (A/B) flash architecture from the start. The device boots from bank A, and when an update is received, it writes the new firmware to bank B while the device continues normal operation. A CRC or SHA hash is verified on the complete image before any switchover. The bootloader checks a valid flag and image integrity on every power-up; if bank A is corrupted, it falls back to bank B, and vice versa. For the update process itself, I'd implement a multi-stage protocol: first, the device authenticates the update source (signed firmware images), then transfers the image in chunks with CRC per chunk, and finally performs a full-image verification. If verification fails, the device stays on the current bank and reports the error. The update should be atomic — either the new firmware is fully verified and the boot flag is set, or nothing changes. For critical safety parameters, I'd keep configuration data in a separate EEPROM or dedicated flash sector that persists across firmware updates. The update mechanism itself would be documented in the device's risk management file, with failure modes analyzed per ISO 14971.

**Possible follow-ups:** How would you test the robustness of your update mechanism? What happens if power is lost during the boot flag write?

## Q4: You're integrating a new sensor into an existing firmware codebase that uses a real-time operating system. The sensor datasheet specifies a complex initialization sequence with precise timing requirements. How would you implement this reliably?

**Answer:** I would encapsulate the sensor initialization in a dedicated driver module with a well-defined API, keeping the complex timing logic isolated from the rest of the system. The initialization sequence would be implemented as a state machine, with each state representing a step in the datasheet's sequence (e.g., power-up, wait for stabilization, send configuration registers, verify register values). Each state transition would include a timeout, so if a step doesn't complete within the specified window, the driver returns an error code rather than hanging indefinitely. For the precise timing requirements, I'd use the RTOS's hardware timer abstraction rather than busy-wait loops — a timer callback or semaphore signals when the next step should execute. This keeps the CPU available for other tasks during the wait periods. I'd also add a retry mechanism with a limited number of attempts, because sensors can sometimes fail initialization due to power supply glitches or bus noise. The driver would expose a status flag indicating whether initialization succeeded, and the application layer would check this before attempting to read data. For debugging, I'd include optional verbose logging that prints each initialization step and its result, which can be enabled during development but compiled out for production.

**Possible follow-ups:** How would you handle the case where the sensor initialization succeeds but then intermittently returns invalid data? What if the sensor requires calibration coefficients that must be loaded at each power-up?

## Q5: Imagine you're leading a firmware team where two engineers disagree on whether to use a state machine pattern or a table-driven approach for implementing a complex device protocol. How would you guide the team to a decision?

**Answer:** This is a design trade-off where both approaches have merit, so I'd facilitate a structured decision process rather than simply picking one. First, I'd ask each engineer to articulate their reasoning with concrete examples from the protocol in question — what specific behaviors are easier or harder to implement, test, and maintain with each approach. I'd then guide the team to evaluate both options against objective criteria: readability for someone new to the codebase, ease of adding new protocol states or commands later, testability (can each state or table entry be unit-tested in isolation?), memory footprint, and worst-case execution time. For a complex protocol, a state machine often makes the control flow explicit and easier to debug, while a table-driven approach can be more compact and data-driven, which is useful if the protocol is likely to change frequently. I might suggest a hybrid: use a state machine for the high-level protocol flow, but table-driven dispatch for command handling within each state. Ultimately, I'd ask the team to prototype a small but representative portion of the protocol in both styles, then review the code together. The goal isn't to declare a winner but to build shared understanding and ownership of the decision. If consensus still doesn't emerge, I'd make the call based on which approach better supports the system's long-term maintainability and testability requirements.

**Possible follow-ups:** How would you handle it if one engineer feels strongly that their preferred approach is the only correct one? What documentation would you expect the team to produce alongside whichever approach is chosen?
