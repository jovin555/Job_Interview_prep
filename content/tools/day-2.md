## Q1: How would you approach selecting a PCB design tool for a new medical device project?

**Answer:** I'd start by mapping the project's specific requirements against tool capabilities. For a medical device, key considerations include: design rule checking (DRC) that can enforce IPC and IEC 60601 creepage/clearance rules, 3D mechanical integration for enclosure fit, and simulation capabilities for signal integrity and thermal analysis. I'd also evaluate the tool's library management system — medical devices require strict component traceability and revision control, so a tool with a centralized, version-controlled component database is preferable over one with loose file-based libraries. Team familiarity matters too: if the team already has deep expertise in Altium Designer or Cadence Allegro, the learning curve cost of switching to KiCad or another tool needs to be weighed against any feature advantages. For a Class II device, I'd lean toward tools that support multi-sheet hierarchical schematics, differential pair routing, and impedance-controlled stackup management, as these are common requirements. Finally, I'd verify the tool can export the documentation formats needed for the Design History File (DHF) — PDF schematics, Gerber/ODB++ fabrication files, and bill of materials in a format compatible with the company's ERP system.

**Possible follow-ups:** How would you handle a situation where your preferred tool isn't available and you must use a less capable one? What specific DRC rules would you set up for IEC 60601 compliance?

## Q2: Walk me through your process for debugging a firmware timing issue using an oscilloscope and logic analyzer together.

**Answer:** I'd start by clearly defining the symptom — is a sensor reading arriving late, a control loop missing its deadline, or a communication bus timing out? Then I'd set up the logic analyzer to capture the relevant digital signals (e.g., SPI chip select, clock, and data lines, or a GPIO toggle that marks the start of a critical ISR). I'd use the oscilloscope on analog signals that might reveal root causes — for example, the power rail feeding the microcontroller, or an analog sensor output that the firmware samples. The key is to trigger both instruments from the same event, often using the logic analyzer's trigger output to arm the oscilloscope, or using a shared trigger signal like a test point toggled by firmware. I'd look for correlations: does a glitch on the power rail coincide with a missed SPI transaction? Does the sensor's analog settling time exceed the firmware's sampling window? I'd also check for interrupt latency by measuring the time between an external event (e.g., a rising edge on an interrupt pin) and the firmware's response (a GPIO toggle inside the ISR). If the issue is intermittent, I'd set up a deep memory capture on the logic analyzer with a pre-trigger buffer, so I can see what happened just before the failure.

**Possible follow-ups:** How would you decide whether to use the oscilloscope or logic analyzer first? What if the timing issue only occurs once every few hours?

## Q3: How would you approach version control for a mixed hardware/firmware project, where PCB design files, firmware source code, and documentation all need to be managed?

**Answer:** I'd advocate for a unified repository structure that keeps hardware and firmware in the same version control system, but with clear separation. For PCB design files (Altium or KiCad project files), Git works well for text-based formats like KiCad's .kicad_sch and .kicad_pcb files, but binary formats like Altium's .SchDoc and .PcbDoc require careful handling — I'd use Git LFS (Large File Storage) for those, and enforce that only one person modifies a PCB project at a time to avoid merge conflicts. Firmware source code lives in the same repo under a separate directory, following standard Git practices with feature branches and pull requests. Documentation — including schematics PDFs, datasheets, and test procedures — goes in a docs folder, with binary files also handled via LFS. The key is to tie hardware and firmware releases together with tags: a v1.0 tag would point to the exact PCB revision, firmware commit, and documentation set that were used for a given build. For medical devices, this traceability is critical for the DHF — if a field failure occurs, you need to know exactly which hardware revision and firmware version were in the affected units. I'd also set up a CI pipeline that runs firmware builds and basic hardware design rule checks on every commit, so issues are caught early.

**Possible follow-ups:** How would you handle a situation where the hardware team uses a different VCS tool than the firmware team? What branching strategy would you recommend for a medical device with multiple variants?

## Q4: Describe how you would use simulation tools to verify a power supply design before building the first prototype.

**Answer:** I'd start with a DC operating point simulation to verify biasing and quiescent conditions — checking that the feedback network sets the correct output voltage, that the error amplifier is in its linear region, and that no component is exceeding its voltage or current ratings. Next, I'd run an AC analysis to examine the control loop's stability: plotting the loop gain's magnitude and phase to check gain margin (typically >10 dB) and phase margin (typically >45° for a robust design, >60° for low-overshoot response). I'd pay special attention to the crossover frequency — it should be well below the switching frequency for a switched-mode supply. Then I'd run a transient simulation with a load step (e.g., 10% to 90% of rated load) to see the output voltage deviation and settling time, and compare against the device's requirements. For a medical device, I'd also simulate startup behavior — does the output ramp up monotonically, and does it stay within the powered device's specified voltage range during startup? I'd include parasitic elements like equivalent series resistance (ESR) of capacitors and trace inductance, as these significantly affect real-world behavior. Finally, I'd run a Monte Carlo analysis with component tolerances to see how manufacturing variation affects output voltage accuracy and stability — this helps identify if the design is marginal and needs tighter tolerance components or a different topology.

**Possible follow-ups:** What would you do if the simulation shows instability but you're not sure which component is causing it? How do you validate that your simulation model accurately represents the real components?

## Q5: Imagine you're leading a design review where a junior engineer presents a PCB layout you believe has significant signal integrity issues. How would you handle this situation constructively?

**Answer:** I'd approach it as a coaching opportunity rather than a critique session. First, I'd acknowledge the work they've put in and ask them to walk me through their design decisions — this helps me understand their thought process and whether they're aware of the trade-offs they've made. For example, if I see a long, unbroken return path under a high-speed trace, I might ask: "Can you describe how the return current flows for this differential pair?" rather than immediately saying "This return path is broken." This encourages them to think through the problem themselves. If they identify the issue, I can reinforce their insight; if not, I'd explain the principle (e.g., "At high frequencies, the return current follows the path of least inductance directly under the signal trace — so this ground plane cut forces it to detour, creating a large loop area that will radiate and pick up noise"). I'd then suggest specific simulation or measurement techniques they could use to verify the concern, such as running a 3D field solver on that section of the layout, or building a small test coupon to measure crosstalk. I'd also offer to sit with them to rework that section of the layout, explaining my routing choices as we go. The goal is to build their understanding so they can avoid similar issues in future designs, not just to fix this one layout. If the project timeline is tight, I'd be clear about which issues are critical (must fix before fabrication) versus nice-to-improve (can be addressed in the next revision), so they learn to prioritize.

**Possible follow-ups:** What if the junior engineer disagrees with your assessment and argues their approach is fine? How would you document the review findings for the project's design history file?
